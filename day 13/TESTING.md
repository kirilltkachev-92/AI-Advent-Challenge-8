# Как тестировать кодинг-агента с автоматом задачи (день 13)

Проверки из задания: **(1) пауза на любом этапе** и **(2) продолжение без повторных объяснений**,
плюс формализованные переходы и многоагентность по этапам. В конце — авто-тесты.

---

## 0. Запуск

```bash
cd "day 13"
cp .env.example .env          # впишите DEEPSEEK_API_KEY
./run.sh
```

При первом запуске пройдите короткое интервью профиля разработчика (язык, стек, уровень,
подробность, ограничение) — оно подмешивается во всех агентов.

### Команды

```
/task <запрос>   начать задачу        /next   принять этап и дальше
/state           состояние задачи     /back   на предыдущий этап
/show <этап>     артефакт этапа        /stage <id>  перейти на этап (валидируется)
```

> Полный проход задачи делает ~10 вызовов LLM. Для быстрой проверки можно ограничиться planning.

## Проверка 1. Многоагентные этапы и артефакты

```
/task CLI-утилита: посчитать частоту слов в файле и вывести топ-N
```
Ожидаемо на этапе **planning**:
```
🤖 Агенты этапа (4):
   рой параллельно:
     • research:architecture — архитектура · … tok
     • research:risks — риски и крайние случаи · … tok
     • research:libraries — библиотеки и API · … tok
   → planner — синтез плана/ТЗ · … tok
── Артефакт: план/ТЗ … ──
⏸ Пауза на этапе «Планирование» …
```
Три исследователя бегут **параллельно**, планировщик сводит результат в артефакт-план.

```
/next      # execution: coder ∥ test-writer → код + тесты
/next      # validation: review:correctness ∥ review:security → validator (ВЕРДИКТ: PASS/FAIL)
/next      # done: orchestrator сводит итог
```
На каждом этапе видно, какие агенты запускались. `/show planning|execution|validation|done` —
показать полный артефакт любого этапа.

## Проверка 2. Детерминированные переходы (в коде)

```
/task что-нибудь
/stage done          # planning → done запрещён
```
→
```
⨯ Переход отклонён: переход «Планирование» → «Готово» запрещён
  Разрешено сейчас: Выполнение (/stage execution)
```
Состояние не меняется. Из `validation` разрешён `/back` в `execution` (вернуть на доработку).

## Проверка 3. Пауза на любом этапе и продолжение без повторов (главная)

После каждого этапа пайплайн **останавливается** сам и ждёт. Это и есть пауза.

**Сессия 1** — дойдите до середины и выйдите:
```
/task сделай функцию факториала с проверкой отрицательных
/next            # planning → execution (код готов)
/exit
```
**Сессия 2** — запустите снова (интервью НЕ появится):
```bash
./run.sh
```
```
/state                 # этап = Выполнение, артефакты planning и execution готовы
/show execution        # код восстановлен с диска — НЕ перегенерирован
/next                  # продолжаем дальше на validation
```
Состояние лежит в файле — откройте и убедитесь:
```bash
cat "day 13/state/task.md"
```
Артефакты сохранены, поэтому следующий этап получает их как вход без повторных объяснений.

## Проверка 4. Замечания и доработка

```
/task сделай парсер CSV
# на этапе планирования не согласны с планом — пишете замечания обычным текстом:
учитывай файлы с разделителем «;» и кавычками
```
Текущий этап (планирование) **перегенерируется** с учётом замечаний (рой агентов запускается
заново). Так же можно дорабатывать код на execution и т.д. `/back` возвращает на прошлый этап.

## Авто-тесты (без обращения к API)

```bash
cd "day 13"
./gradlew test
```
13 тестов: таблица переходов, прямой путь и `/back`, отклонение недопустимого перехода без
изменения состояния, артефакты по этапам, snapshot/restore (пауза/возобновление); planning
поднимает рой из 3 исследователей + планировщик, execution — coder ∥ test-writer, validation —
рой из 2 ревьюеров + валидатор, профиль подмешивается во всех агентов; `startTask` прогоняет
planning, `advance` запускает следующий этап, недопустимый переход блокируется, состояние
переживает перезапуск.
