# День 14 — инварианты и ограничения состояния

Кодинг-агент из дня 13 + **инварианты**: жёсткие ограничения, которые ассистент НЕ имеет права
нарушать ни при каких условиях (выбранная архитектура, принятые решения, ограничения по стеку,
бизнес-правила). Инварианты хранятся отдельно от диалога, подмешиваются в каждого агента и
проверяются двойным контролем (код + LLM). При конфликте запроса с инвариантом агент **отказывается**
и объясняет почему.

Дизайн следует разбору наставника (Алексей Гладков): инварианты — это «подкапотные константные
ограничения» и «самозапреты» пользователя, которые он может попытаться обойти, а агент не даст;
проверка — **и детерминированная в коде, и через LLM** («два разных способа, не взаимоисключают»),
вплоть до нескольких контролёров.

## Что такое инвариант

```
Invariant(id, category, rule, forbid)
  category — стек | решение | архитектура | бизнес-правило | пользовательский
  rule     — человекочитаемое правило (для промпта и LLM-контролёра)
  forbid   — запрещённые токены для детерминированной проверки в коде (необязательно)
```

Инварианты лежат **отдельно от диалога и задачи** — в `invariants/invariants.json`. При первом
запуске подсевается набор примеров:

| Категория | Правило | Проверка |
|-----------|---------|----------|
| стек | RxJava запрещена (только Coroutines/Flow) | код (`rxjava`, `io.reactivex`) + LLM |
| архитектура | только MVVM/MVI, MVP запрещён | код (`mvp`) + LLM |
| решение | только OSS-зависимости (MIT/Apache), без GPL и платных SDK | код (`gpl`) + LLM |
| архитектура | без бизнес-логики в UI-слое | LLM |
| бизнес-правило | секреты не хранить в коде | LLM |

## Как соблюдаются (три механизма)

1. **Явный учёт в рассуждениях.** Блок инвариантов добавляется в system prompt **каждого** агента
   пайплайна (research/planner/coder/reviewer/validator) — они обязаны их учитывать.
2. **Отказ на конфликтный запрос (pre-flight).** Перед стартом задачи запрос проверяется
   контролёром. Если он требует нарушить инвариант — задача **не берётся** (`⛔ Отказ`), агент
   объясняет, какой инвариант нарушен, и предлагает допустимую альтернативу.
3. **Принуждение на этапе кода (post-execution).** Сгенерированный код проверяется на инварианты;
   при нарушении этап выполнения переделывается с явным фидбэком (до 2 попыток). Если нарушение
   остаётся — оно показывается как предупреждение (`⚠`), а не выдаётся за корректный результат.

## Двойная проверка (код + LLM)

`InvariantChecker` объединяет:
- **детерминированную** проверку по `forbid`-токенам (регистронезависимо, по границам слов —
  «java» сматчит `Java 17`, но не `javascript`; защита от ложных срабатываний на подстроках);
- **семантическую** через LLM-контролёра (ловит нарушения, которые подстрокой не найти,
  например «бизнес-логика во вью»).

Находки объединяются с дедупликацией по инварианту. Способы дополняют друг друга.

## Команды

```
/inv                  показать инварианты (и где хранятся)
/inv add <правило>    добавить инвариант (проверяется LLM-контролёром)
/inv forbid <токен>   запретить токен (детерминированная проверка) — «самозапрет»
/inv rm <n>           удалить инвариант №n
/inv clear            удалить все
/check <текст>        проверить произвольный текст на инварианты (код + LLM)
```
Плюс команды дня 13: `/task`, `/state`, `/show`, `/next`, `/back`, `/stage`, `/profile`, `/demo`,
`/reset`. Полный список — `/help`.

`/demo` показывает инварианты на задаче «Android Hello World»: сначала конфликтный запрос
(Hello World на **MVP + RxJava**) → агент отказывает, отмечая оба нарушения и объясняя их; затем
тот же продукт без запрещённого → пайплайн принимает запрос и собирает код в рамках инвариантов.

## Проверки из задания

- **Конфликт запроса и инварианта:** `/task Добавь RxJava …` → агент отказывается, задача не
  стартует.
- **Как объясняет отказ:** ссылается на конкретный нарушенный инвариант (категория + правило),
  показывает, чем именно нарушено и каким контролёром поймано, и предлагает совместимую
  альтернативу.
