# День 33. Ассистент поддержки пользователей

**Мини-сервис поддержки**: консоль оператора, в которой AI-ассистент отвечает
на вопросы о продукте по **RAG (FAQ + документация)** и **учитывает контекст
пользователя и тикета**, доставая его из **CRM через MCP**.

Продукт и данные — вымышленные, сгенерированные, как и советовали в чате
(«crm можно на коленке наваять, можно взять готовую»; «у тебя есть ЛЛМ —
Создай!»): таск-трекер **«Планёрка»** с базой знаний из 5 документов
(`data/docs`) и CRM-JSON с 5 пользователями и 7 тикетами (`data/crm.json`).

## Как устроено

```
вопрос оператора/пользователя
    │
    ├─ RAG: Ollama nomic-embed-text ──top-K──▶ фрагменты FAQ и документации
    │        (data/docs → output/index-kb.json, кеш по SHA корпуса)
    │
    ├─ контекст обращения: карточка пользователя и/или активный тикет
    │        (команды /user и /ticket — тоже ходят в CRM через MCP)
    │
    ▼
DeepSeek (function calling) ◀──── tools/list ──── CRM MCP-сервер (localhost:8033)
    │        tools/call: find_user · list_tickets ·        │
    │                    get_ticket · add_ticket_note      ▼
    │                                        data/crm.json → output/crm-state.json
    ▼                                        (исходник в git, изменения отдельно)
ответ + вызовы MCP + источники RAG
```

- **CRM через MCP** (`CrmMcp.kt`, `McpServer.kt`): свой MCP-сервер, как в Днях
  17 и 31 — JSON-RPC 2.0 поверх Streamable HTTP, вручную. Ассистент видит
  данные пользователей и тикетов только через инструменты; `add_ticket_note`
  пишет в CRM (изменения сохраняются в `output/crm-state.json`, исходный
  `data/crm.json` не трогается — удалить state-файл = сбросить CRM).
- **RAG** (`KnowledgeBase.kt`, `Indexer.kt`, `Search.kt`): markdown-разделы
  базы знаний → эмбеддинги локальной Ollama (`nomic-embed-text`), гибридный
  поиск (косинус + бонусы за точные слова), развёртка соседними чанками —
  весь конвейер Дней 28–32.
- **Контекст пользователя/тикета** (`SupportAgent.kt`, `Main.kt`): оператор
  выбирает `/user marina` — карточка из CRM попадает в каждый запрос, и промпт
  требует сначала посмотреть тикеты этого человека. На вопросе из задания
  «Почему не работает авторизация?» ассистент сам вызывает
  `list_tickets` → `get_ticket`, видит в тикете T-1042 ошибку AUTH-403 после
  сброса пароля и отвечает по `auth.md` про 30-минутную блокировку — а не
  генеричной инструкцией.
- Протоколы вручную, как весь марафон: клиенты DeepSeek, Ollama и MCP —
  `java.net.http`, сервер — `com.sun.net.httpserver`. Без SDK.

## Запуск

```bash
cd "day 33"
./run.sh            # консоль оператора (нужны Ollama + DEEPSEEK_API_KEY)
./run.sh index      # только построить индекс базы знаний
./run.sh report     # чеклист задания без интерактива → output/report.md
./demo.sh           # скриптованная сессия для видео
```

Команды REPL: `/user <email|имя>`, `/ticket <id>`, `/tickets`, `/reset`,
любой другой текст — вопрос ассистенту.

## Чеклист задания

- ✅ **отвечает на вопросы о продукте** — тарифы, авторизация, интеграции;
  не по продукту — вежливый отказ, нет в базе — честное «не знаю»
- ✅ **использует RAG (FAQ, документация)** — 5 документов → 30 чанков,
  источники печатаются под каждым ответом
- ✅ **учитывает контекст пользователя или тикета** — `/user` и `/ticket`
  кладут карточку и переписку в запрос, остальное ассистент сам достаёт из CRM
- ✅ **через MCP подключён JSON с пользователями/тикетами** — свой MCP-сервер,
  4 инструмента, включая запись заметки в тикет
- ✅ **пример из задания** — «Почему не работает авторизация?» отвечается
  с учётом данных тикета T-1042 (см. отчёт, случай 2)

Прогон чеклиста с реальными вызовами — [`output/report.md`](output/report.md).
