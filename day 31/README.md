# День 31. Ассистент разработчика

**Ассистент, понимающий проект**: RAG по документации репозитория (README всех
31 дня + [`docs/`](docs) со схемой данных и API-описанием) + собственный
MCP-сервер с git-инструментами. Команда `/help` отвечает на вопросы о структуре
и устройстве проекта, а про живое состояние репозитория (ветка, файлы, diff,
коммиты) модель сама зовёт git через MCP.

По обсуждению в чате: тьютор подтвердил, что с этой недели снова можно облачную
модель и что «задания конкретные — нужно именно с гитом поработать» — поэтому
генерация здесь DeepSeek, а все MCP-инструменты — обёртки над настоящими
командами git.

## Как устроено

```
day */README.md + day 31/docs/*.md ──чанки по разделам──▶ nomic-embed-text ──▶
        │                                                  output/index-docs.json
вопрос ──/api/embed──▶ вектор ──cosine+keywords──▶ top-5 фрагментов ─┐
                                                                     ▼
клиент ──/help──▶ DeepSeek (function calling) ◀──tools/list── MCP git-сервер
                        │ tool_calls                (localhost:8031, JSON-RPC 2.0)
                        └──tools/call──▶ git_branch / git_status / git_files /
                                          git_diff / git_log  (настоящий git)
```

- **RAG** (`DocsCorpus.kt`, `Indexer.kt`, `Search.kt`): корпус — README каждого
  дня плюс `docs/` (архитектура, [API MCP-инструментов](docs/mcp-tools.md),
  [схема данных индекса](docs/index-format.md)). Один чанк = один раздел
  markdown; индекс в формате Недели 5, кеш по SHA-256 корпуса. Поиск —
  косинус + бонусы за точные слова и биграммы.
- **MCP** (`McpServer.kt`, `GitMcp.kt`): свой сервер, JSON-RPC 2.0 поверх
  Streamable HTTP — код Дня 17, инструменты другие: `git_branch` (минимум по
  заданию), `git_status`, `git_files`, `git_diff`, `git_log`. Ассистент
  подключается к нему по-настоящему — `initialize` → `tools/list` →
  `tools/call` через `McpClient.kt` (День 16/17).
- **Ассистент** (`Assistant.kt`): DeepSeek с function calling. Фрагменты
  документации кладутся в промпт, git-инструменты отдаются как функции;
  системный промпт запрещает отвечать про состояние репозитория по памяти.
  Под каждым ответом — реальные вызовы MCP и файлы-источники RAG.
- Протоколы вручную, как весь марафон: `com.sun.net.httpserver` из JDK и
  `java.net.http`, никаких фреймворков и SDK.

## Команды REPL

| Команда | Что делает |
|---|---|
| `/help <вопрос>` | вопрос о проекте: RAG по документации + git через MCP |
| `/branch` `/status` `/log` | git-инструменты MCP напрямую |
| `/files [dir]` `/diff [путь]` | список файлов / diff (опционально по пути) |
| `/quit` | выход |

Текст без команды тоже трактуется как вопрос о проекте.

## Чеклист задания

Режим `report` гоняет контрольные вопросы через тот же пайплайн, что REPL, —
отчёт в [`output/report.md`](output/report.md):

- ✅ **README + папка docs в RAG** — 30+ README и `docs/` в индексе,
  под каждым ответом видны файлы-источники
- ✅ **MCP — хотя бы git branch** — `git_branch` плюс статус/файлы/diff/лог,
  живой вызов через `tools/call` в отчёте
- ✅ **/help отвечает на вопросы о структуре проекта** — включая
  вопрос-ловушку, на который в документации ответа нет
