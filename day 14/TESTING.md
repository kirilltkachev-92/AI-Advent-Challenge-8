# Как тестировать инварианты (день 14)

Проверки из задания: **(1) что происходит при конфликте запроса и инварианта** и
**(2) как ассистент объясняет отказ**, плюс хранение отдельно от диалога и двойная проверка.

---

## 0. Запуск

```bash
cd "day 14"
cp .env.example .env          # впишите DEEPSEEK_API_KEY
./run.sh
```
При первом запуске пройдите короткое интервью профиля (свободные ответы). Инварианты
подсеваются автоматически.

---

## Проверка 1. Инварианты хранятся отдельно и видны

```
/inv
```
Показывает список инвариантов и путь `invariants/invariants.json`. Откройте файл — он отдельно
от диалога и состояния задачи:
```bash
cat "day 14/invariants/invariants.json"
```

---

## Проверка 2. Конфликт запроса и инварианта → отказ с объяснением (главная)

Инвариант по умолчанию: «RxJava запрещена». Дайте конфликтный запрос:
```
/task Добавь зависимость RxJava для реактивных потоков
```
Ожидаемо:
```
⛔ Отказ: запрос конфликтует с инвариантами проекта.
   • [стек] Реактивность только на Kotlin Coroutines/Flow. RxJava запрещена.
     нарушение: найдено «RxJava» · проверка: код

Объяснение:
Не могу выполнить … нарушает инвариант … Допустимая альтернатива: Coroutines/Flow …
```
Задача **не стартует** (`/state` покажет, что активной задачи нет). Отказ ссылается на конкретный
инвариант, показывает, чем нарушено и каким контролёром (код/llm) поймано, и предлагает
альтернативу.

Семантический конфликт (ловит LLM, без forbid-токена):
```
/task Сложи всю бизнес-логику прямо во вью, без слоёв
```
→ отказ по инварианту «без бизнес-логики в UI-слое».

## Проверка 3. Двойная проверка: код + LLM (`/check`)

```
/check import io.reactivex.Flowable        # поймает детерминированная проверка (код)
/check храните пароль прямо в коде: val p = "123"   # поймает LLM-контролёр
/check обычная функция на корутинах        # ✓ нарушений нет
```

Защита от ложных срабатываний на подстроках:
```
/inv forbid java
/check пишем на javascript                 # ✓ нет (граница слова)
/check берём Java 17                        # ⛔ нарушение
/inv rm <номер этого правила>
```

## Проверка 4. Самозапреты пользователя

```
/inv forbid Spring          # детерминированный запрет токена
/inv add Не использовать глобальные синглтоны для состояния   # правило для LLM
/inv
/task Сделай сервис на Spring Boot          # ⛔ отказ по вашему запрету
/inv rm 5
```

---

## Проверка 5. Принуждение на этапе кода

Если код, сгенерированный на этапе execution, нарушает инвариант, агент сам переделывает этап
(до 2 попыток), а если нарушение остаётся — показывает предупреждение и не выдаёт результат за
корректный:
```
... начните задачу, дойдите до execution ...
⚠ Код всё ещё нарушает инварианты после попыток исправления: …
```
Можно дать замечание текстом или `/back` для переделки.
