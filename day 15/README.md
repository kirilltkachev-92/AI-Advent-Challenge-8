# День 15 — контролируемые переходы состояний

Android кодинг-агент из дней 13–14 с **контролируемым жизненным циклом задачи**: у задачи есть
явные состояния, разрешён только фиксированный набор переходов между ними, и ассистент **не может
«перепрыгнуть» этап**. Плюс по совету наставника на этапе планирования работает **совет агентов с
оркестратором**, который сводит мнения и возражает.

## Жизненный цикл и разрешённые переходы

Базовый набор стадий из конспекта недели 3 (clarify → planning → execution → validation → done):

```
 clarify ─/next▶ planning ─/next▶ execution ─/next▶ validation ─/next▶ done
    ▲               ▲                  │                  │
    └────/back──────┴──────/back───────┴──────/back───────┘
```

Таблица переходов — единственный источник истины (`TaskStage.transitions`, проверяется в коде):

| Этап | Куда можно |
|------|-----------|
| `clarify` | `planning` |
| `planning` | `execution`, `clarify` |
| `execution` | `validation`, `planning` |
| `validation` | `done`, `execution` |
| `done` | — |

Любой другой переход отклоняется. Это прямо реализует правила из задания:
- **нельзя планировать без уточнённых требований** — в `planning` можно попасть только из `clarify`;
- **нельзя реализацию без утверждённого плана** — в `execution` можно попасть только из `planning`
  (переход `/next` = утверждение плана пользователем), и только при наличии артефакта-плана;
- **нельзя финал без валидации** — в `done` можно попасть только из `validation`.

## Три механизма контроля (всё в коде)

1. **Таблица переходов.** `/next`, `/back`, `/stage <id>` идут через валидацию; недопустимый
   переход не меняет состояние и показывает допустимые варианты.
2. **Предусловие этапа.** Этап нельзя начать без артефакта предыдущего: `moveTo` проверяет, что
   результат предшествующего этапа существует («сначала нужен результат этапа …»).
3. **Инвалидация вниз по цепочке.** Перегенерация раннего этапа стирает артефакты последующих —
   после повторного планирования снова нужно пройти execution и validation, «перепрыгнуть» вперёд
   по устаревшим данным нельзя.

Плюс **детектор попыток обойти этапы обычным текстом**: фразы вроде «пропусти план, сразу пиши код»
распознаются в коде и получают отказ с объяснением жизненного цикла (а не выполняются как замечание).

## Совет агентов с дебатами + оркестратор (planning)

По совету наставника этап планирования — это не один агент, а совет, который **обменивается
мнениями** в два раунда:

```
planning:  proposer[architecture|risks|libraries|ui-structure] ∥     (раунд 1 — предложения)
        →  debater[architecture|risks|libraries|ui-structure]  ∥     (раунд 2 — дебаты)
        →  orchestrator                                               (сводит, возражает, решает)
```

- **Раунд 1:** четыре агента параллельно дают независимые предложения с разных сторон.
- **Раунд 2 (дебаты):** каждый агент видит предложения коллег и явно возражает/соглашается,
  указывает конфликты и корректирует своё — это и есть «обмен мнениями».
- **Оркестратор** знает исходный запрос, учитывает исход дебатов, разрешает конфликты, сводит всё
  в единый план-ТЗ и **возражает** (блок «Возражения и решения совета»).

Перед планированием идёт этап **clarify**: агент-аналитик превращает запрос в уточнённые
требования (цель, функциональные/нефункциональные требования, допущения, критерии приёмки),
которые и поступают на вход совету. Стейдж-агенты изолированы: получают только нужный вход
(чёткие контракты), а не всю историю.

## Пауза и продолжение

Состояние (этап + артефакты) сохраняется в `state/task.md`. Работу можно прервать на любом этапе;
при следующем запуске `/state` покажет тот же этап и готовые артефакты, а переходы продолжают
работать по тем же правилам (например, из восстановленного `execution` можно только в `validation`
или назад в `planning`).

## Проверки из задания

- **Попытка перейти в недопустимое состояние:** `/stage done` из `clarify`/`planning` → отклонено;
  то же для обхода текстом.
- **Реакция ассистента:** сообщает, что этапы перепрыгивать нельзя, показывает текущий этап и
  допустимые переходы, напоминает правила (нельзя план без требований, реализацию без плана, финал без валидации).
- **Корректность продолжения после паузы:** после перезапуска этап и артефакты восстановлены,
  переходы по-прежнему ограничены.

## Команды

```
/task <запрос>   начать задачу            /state   этап, допустимые переходы, артефакты
/next            принять этап → дальше     /back    вернуться на предыдущий этап
/stage <id>      перейти на этап (валидируется)
```
Плюс `/inv`, `/check`, `/profile`, `/demo`, `/reset` и др. — полный список `/help`.
`/demo` — полное демо всех возможностей: инварианты + отказ по конфликтному запросу (с объяснением)
→ clarify (аналитик) → planning с советом и дебатами → попытки перепрыгнуть этап (командой и текстом) отклоняются →
корректный проход по этапам до `done` → состояние сохраняется (пауза/возобновление).
