# День 26. Запуск локальной LLM

Локальная LLM через **Ollama 0.31.1** (`http://localhost:11434`): проверяем, что сервер
запущен, модель отвечает через CLI и HTTP API, и гоним **4 запроса разной сложности**
(минимум был 3) — причём на двух моделях сразу, большой `qwen2.5:14b` и крошечной
`qwen2.5:0.5b`, чтобы увидеть цену «локальности» в качестве и скорости.

## Как устроено

- **Протокол Ollama написан вручную** (`OllamaClient.kt`) поверх `java.net.http.HttpClient`,
  как и в прошлые дни — никаких SDK, весь обмен виден: `GET /api/version` (сервер жив),
  `GET /api/tags` (модели скачаны), `POST /api/chat` со `stream=false` и `temperature=0`
  (прогон воспроизводим).
- Ollama в ответе отдаёт метрики — `prompt_eval_count`/`eval_count` (токены) и
  `*_duration` (наносекунды): из них считаем скорость генерации и латентность,
  ничего не замеряя секундомером «снаружи».
- **Каждый ответ проверяется кодом** (`Prompts.kt`): не «модель что-то сказала»,
  а честный ✓/✗ — факт сверяется со строкой, у логической задачи парсится
  «Ответ: N», JSON реально парсится и проверяются поля и дата.
- Прогон пишет отчёт с полными ответами и метриками в [`output/report.md`](output/report.md).

## 4 запроса по нарастающей сложности

1. **Простой факт** — «Столица Австралии?» (с подвохом: частый неверный ответ — Сидней).
2. **Генерация кода** — `isPalindrome` на Kotlin, игнорируя регистр и пунктуацию.
3. **Логическая ловушка** — вариант bat-and-ball: «ручка и карандаш стоят 110 ₽,
   ручка дороже на 100 ₽» (интуитивный неверный ответ — 10, правильный — 5).
4. **Структурированное извлечение** — JSON с 5 полями из русского текста,
   включая нормализацию даты «12 марта 2026» → ISO 8601.

## Результаты (полный прогон: `./run.sh`)

| Запрос | qwen2.5:14b (9 ГБ) | qwen2.5:0.5b (0.4 ГБ) |
|---|---|---|
| 1 — простой факт | ✓ «Канберра», 0.3 с | ✗ «Монтagu» (!) |
| 2 — генерация кода | ✓ корректный код, 1.7 с | ✓* структура есть, логика неверна |
| 3 — логическая ловушка | ✓ «Ответ: 5 рублей», 10.2 с | ✗ «Ответ: 10 рублей» — попалась |
| 4 — извлечение JSON | ✓ все поля, дата ISO, 2.2 с | ✓ все поля, дата ISO |
| скорость генерации | ~25 ток/с | ~260 ток/с |

\* формальную проверку (есть функция, есть тип) прошла, но код неверный: строку
не чистит и не приводит к нижнему регистру — хорошее напоминание, что shallow-проверки
кода мало.

Наблюдения:

- **14B прошла всё, включая ловушки**: не сказала «Сидней», решила bat-and-ball
  через систему уравнений, отдала чистый JSON без ограды и пояснений.
- **0.5B — наглядная цена компактности**: в 10 раз быстрее (~260 против ~25 ток/с),
  но галлюцинирует на фактах («Монтagu» вместо Канберры) и берёт интуитивный
  неверный ответ в логической задаче. При этом простое структурированное извлечение
  (JSON из текста) ей вполне по силам — ниша маленьких моделей именно такая.
- **Всё честно локально**: после выгрузки в офлайне работает так же — модель на диске
  (`ollama list`), запросы не уходят дальше `localhost:11434`.

## Запуск

```bash
brew install ollama && ollama serve   # если ещё не запущена
ollama pull qwen2.5:14b               # основная модель (~9 ГБ)
ollama pull qwen2.5:0.5b              # маленькая для сравнения (~400 МБ)

./run.sh    # health-check + 4 запроса × 2 модели → output/report.md
./demo.sh   # демо для видео: CLI (ollama run), HTTP (curl), полный прогон
```

Проверка руками, без Kotlin:

```bash
ollama run qwen2.5:14b "Что такое локальная LLM?"          # CLI
curl -s http://localhost:11434/api/chat -d '{
  "model": "qwen2.5:14b",
  "messages": [{"role": "user", "content": "2+2?"}],
  "stream": false
}'                                                          # HTTP API
```
