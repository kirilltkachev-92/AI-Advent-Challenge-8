# День 35. Реальная задача: AI-конвейер подготовки релиза

**Какую задачу решаю.** Челлендж закончился, и у репозитория из 35 дней и 40+
коммитов нет ни одного тега и релиза. Хочется «подготовить релиз» по-взрослому,
но руками это муторно: перечитать всю историю, сгруппировать месяц работы в
внятные release notes, проверить, что ничего не забыто и не расходится с origin,
завести тег и оформить GitHub Release. Этот конвейер делает всё это одной
командой — **и публикует настоящий релиз v1.0.0 этого самого репозитория**.

**Как AI участвует.** Модель делает две вещи, которые вручную дольше всего:
1. **Пишет release notes** — DeepSeek получает реальную историю коммитов и темы
   всех 35 дней (первые строки README каждого дня), группирует марафон по темам,
   выбирает самые интересные дни и предлагает semver-версию с обоснованием.
2. **Release gate** — по фактам репозитория и результатам преflight-проверок
   выносит вердикт GO/NO-GO с блокерами и предупреждениями (например, отличает
   мусорные `.DS_Store` от реально незакоммиченных изменений).

Жёсткие проверки при этом остаются детерминированными: публикацию блокируют
только FAIL-ы преflight, «переубедить» кодовую часть модель не может.

## Как устроено

```
git / gh (реальное окружение)
    │  история, статус, ahead/behind, темы дней, auth
    ▼
[1] FactCollector ──► RepoFacts
    ▼
[2] Preflight — детерминированно: ветка main, чистое дерево,
    синхронизация с origin, gh авторизован, есть коммиты
    ▼
[3] DeepSeek: черновик релиза ──► версия + обоснование + release notes (JSON)
    ▼
[4] DeepSeek: release gate ──► GO / NO-GO + блокеры + предупреждения (JSON)
    ▼
[5] Публикация (только publish, только без блокеров):
    git tag -a → git push тега → gh release create --notes-file
```

- **Реальное окружение**: конвейер работает не с макетом, а с этим git-репо и
  настоящим GitHub через `gh` (`Facts.kt`, `Publisher.kt`) — git и gh дёргаются
  как процессы, без библиотек.
- **AI-часть** (`ReleaseAgent.kt`): два вызова DeepSeek со строгим JSON-ответом
  (`response_format: json_object`), протокол вручную, как весь марафон.
- **Безопасность публикации** (`Main.kt`): сухой прогон по умолчанию; publish
  останавливается при FAIL-ах преflight, NO-GO от gate или существующем теге.
- **Артефакты**: `output/release-notes.md` (их же получает GitHub Release) и
  `output/report.md` — факты, таблица проверок, решение модели, итог.

## Запуск

```bash
cd "day 35"
./run.sh            # prepare: сухой прогон без публикации (нужен DEEPSEEK_API_KEY)
./run.sh publish    # то же + реальный тег и GitHub Release (нужен gh auth)
./demo.sh           # скриптованная сессия для видео: prepare + publish
```

### Повторный прогон публикации

Релиз настоящий, поэтому второй `publish` честно остановится на «тег уже
существует». Снять релиз для повторной записи демо:

```bash
gh release delete v1.0.0 -y
git push --delete origin v1.0.0
git tag -d v1.0.0
```
