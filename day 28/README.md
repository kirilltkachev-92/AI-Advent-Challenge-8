# День 28. Локальная LLM + RAG

**RAG-система, полностью работающая локально**: пайплайн Недели 5 (чанки →
эмбеддинги → JSON-индекс → top-K поиск), но корпус — **весь чат AI Advent
Challenge #8** (31.05–08.07, ~3700 сообщений), retrieval и генерация — локальные
(`nomic-embed-text` + `qwen2.5:14b` через Ollama). Для сравнения тот же промпт
с теми же фрагментами уходит облачному DeepSeek, а слепой судья оценивает
качество; скорость и стабильность меряются на 3 прогонах каждого вопроса.

Про выбор корпуса: в задании «индекс из Недели 6» — опечатка; тьютор уточнил,
что имеется в виду индекс прошлой недели, и «так-то можно любой брать» —
поэтому здесь проиндексирован весь чат челленджа целиком.

## Как устроено

```
data/messages*.html ──парсер──▶ ~3700 сообщений ──1 сообщение = 1 чанк──▶
        │ /api/embed (nomic-embed-text)                 output/index-chat.json
вопрос ──/api/embed──▶ вектор ──cosine+keywords──▶ top-5 ──▶ фрагменты с контекстом
        │                                                    соседних реплик
system + фрагменты + вопрос ──┬── /api/chat (qwen2.5:14b, локально)
                              └── DeepSeek chat/completions (облако, если есть ключ)
оба ответа ──▶ слепой судья DeepSeek (0–2 против эталона) ──▶ output/report.md
```

- **Протокол Ollama и DeepSeek вручную** (`OllamaClient.kt`, `DeepSeekClient.kt`),
  парсер экспорта Telegram тоже (`TelegramExport.kt`) — никаких SDK.
- **Индекс в формате Недели 5** (`IndexStore.kt`): те же поля `chunk_id/source/
  strategy/embedding`, стратегия `message` — одно сообщение = один вектор,
  а при выдаче фрагмент разворачивается контекстом соседних реплик (`Search.kt`).
  Кеш по SHA-256 корпуса: пересчёт только при смене экспорта.
- **Честное сравнение** (`Comparison.kt`): retrieval один и тот же, промпт
  идентичный, temperature 0 у обеих; судья не знает, чей ответ оценивает.
  Без `DEEPSEEK_API_KEY` прогон остаётся полностью локальным.
- **Контрольный набор** (`ControlSet.kt`): 8 вопросов по чату с зафиксированными
  ожиданиями, включая вопрос-ловушку, ответа на который в чате нет.

## Итоги прогона (полный отчёт — [`output/report.md`](output/report.md))

| метрика | локальная `qwen2.5:14b` | облачная `deepseek-chat` |
|---|---|---|
| качество (слепой судья, макс 16) | **14** | **15** |
| латентность средняя | 8.8 с | 1.5 с |
| скорость генерации | 24.9 ток/с | 52.4 ток/с |
| ошибок (из 24 запросов) | 0 | 0 |
| вопросов, где все 3 ответа идентичны | **8/8** | 3/8 |

Выводы: по качеству локальная почти догнала облако (14 против 15 — на одном
вопросе ответила менее полно), по скорости проигрывает в ~6 раз, зато по
стабильности выигрывает всухую — при temperature 0 локальная модель побайтово
воспроизводима, облачная — нет. На вопросе-ловушке обе честно ответили
«в чате этого нет».

## Чеклист задания

- ✅ используйте индекс из Недели 5 — тот же формат и пайплайн, корпус любой
  (уточнение тьютора); здесь — весь чат челленджа
- ✅ retrieval выполняется локально — `nomic-embed-text` + косинус, всё на
  `localhost:11434`
- ✅ генерация — через локальную модель — `qwen2.5:14b` через `/api/chat`
- ✅ сравните ответы локальной и облачной — идентичный промпт, DeepSeek
- ✅ оцените качество / скорость / стабильность — слепой судья, счётчики Ollama,
  3 прогона каждого вопроса
- ✅ RAG-система, полностью работающая локально — `ASK="вопрос" ./run.sh`
  работает без единого облачного запроса

## Запуск

```bash
brew install ollama && ollama serve   # если ещё не запущена
ollama pull qwen2.5:14b               # генерация (~9 ГБ)
ollama pull nomic-embed-text          # эмбеддинги (~0.3 ГБ)

cd "day 28"
ASK="Какое задание дали в День 28?" ./run.sh   # одиночный вопрос, полностью локально
./run.sh                                        # полное сравнение → output/report.md
./demo.sh                                       # демо для видео
```

Облачная сторона и судья включаются, если найден `DEEPSEEK_API_KEY`
(см. `.env.example`; ключ ищется и в `.env` дней Недели 5). Первый прогон
строит индекс (~3700 эмбеддингов, несколько минут), дальше — из кеша.
