# День 30. Локальная LLM как приватный сервис

**Локальная модель, выложенная наружу как приватный AI-сервис**: HTTP API +
веб-чат поверх Ollama, доступ по API-токену, rate limit, max context и очередь
генераций. Знание сервиса — **весь чат AI Advent Challenge #8** (31.05–11.07,
~3900 сообщений): RAG-пайплайн Дня 28 отвечает на вопросы по истории марафона —
задания, уточнения тьютора, находки участников.

По обсуждению в чате: тьютор уточнил, что смысл дня — «вы делали для себя,
а теперь нужно **выложить для всех**» и «в идеале чтобы я **снаружи** тоже мог
постучаться»; участники выкладывали сервисы с токеном/логином — поэтому здесь
авторизация по ключу обязательна (без `API_TOKENS` сервис не стартует).
Модель по умолчанию — `qwen2.5:1.5b`: в чате проверено, что такая крутится
даже на VPS с 1 vCPU / 2 ГБ RAM без GPU.

## Как устроено

```
data/messages*.html ──парсер──▶ 3913 сообщений ──nomic-embed-text──▶ output/index-chat.json
                                                                     (пересчёт по SHA корпуса)
клиент ──▶ HTTP-слой (com.sun.net.httpserver, без фреймворков)
            401 без токена → 429 rate limit → 413 длинный вход → 503 очередь полна
            └─▶ RAG: вопрос → top-K + соседние реплики → подрезка под max context
                 └─▶ Ollama /api/chat (num_ctx, num_predict) ──▶ ответ + источники + метрики
```

- **Сервер stateless**: историю диалога присылает клиент, на сервере ничего
  не хранится. Веб-чат (`GET /`) — одна страница без зависимостей, история
  в браузере, токен в localStorage.
- **Оборона по порядку дешевизны** (`HttpApi.kt`): сначала 401 (нет токена),
  потом 429 (token bucket на каждый токен: залп 5, темп 12/мин, честный
  Retry-After), потом 413 (сообщение длиннее max input), и только потом запрос
  встаёт в очередь к модели (≤2 генераций разом, ≥9-й ожидающий получает 503) —
  дешёвый отказ дешевле дорогой генерации.
- **Max context с двух сторон** (`ChatService.kt`): модели передаётся жёсткий
  `num_ctx`/`num_predict`, а промпт заранее подрезается бюджетами — история
  с головы (старые реплики уходят первыми, клиенту возвращается
  `history_trimmed: true`), RAG-фрагменты с хвоста списка релевантности.
- **Ограничения — не секрет**: `GET /v1/limits` отдаёт действующие лимиты,
  и режим `verify` проверяет сервис по ним, а не по локальному конфигу.
- Протоколы вручную, как весь марафон: сервер на `com.sun.net.httpserver`
  из JDK, клиент Ollama на `java.net.http`, никаких фреймворков и SDK.

## API

| Маршрут | Токен | Что делает |
|---|---|---|
| `GET /` | — | веб-чат |
| `GET /healthz` | — | статус, модель, uptime — проверка доступа по сети |
| `GET /v1/limits` | ✔ | действующие ограничения сервиса |
| `POST /v1/chat` | ✔ | `{"message":"…"}` или `{"messages":[{role,content}…]}` → ответ, источники, usage, тайминги |

```bash
curl -s http://<хост>:8030/v1/chat -X POST \
  -H "Authorization: Bearer <токен>" -H "Content-Type: application/json" \
  -d '{"message":"Что было в задании дня 21?"}'
```

## Чеклист задания

Проверяет внешний HTTP-клиент — режим `verify` (`Verify.kt`), сервис для него
чёрный ящик; отчёт — [`output/report.md`](output/report.md):

- ✅ **доступ к модели по сети** — bind на `0.0.0.0`, `healthz` + реальный ответ
  модели снаружи процесса; при деплое тот же `verify` гоняется против VPS
  (`VERIFY_BASE_URL=http://<vps>:8030 VERIFY_TOKEN=… ./run.sh verify`)
- ✅ **возможность чата** — веб-чат на `GET /` и multi-turn `messages` в API
- ✅ **стабильность при нескольких запросах** — серия последовательных
  и параллельный залп: под потолком одновременности хвост ждёт в очереди,
  перегруз — это честные 429/503, а не падение или зависание
- ✅ **rate limit** — залп упирается в 429 с `Retry-After`
- ✅ **max context** — сообщение длиннее лимита → 413; длинная история
  подрезается (`history_trimmed: true`), а не роняет сервис
- ✅ **приватность** — без токена / с неверным токеном → 401

## Запуск локально

```bash
ollama pull qwen2.5:1.5b && ollama pull nomic-embed-text

cd "day 30"
cp .env.example .env    # и поменять API_TOKENS
./run.sh                # первый старт построит индекс чата (~40 с) и поднимет сервис
./run.sh verify         # чеклист задания → output/report.md
./demo.sh               # демо для видео (сервис должен уже работать)
```

## Деплой на VPS

```bash
ssh root@<vps> 'bash -s' < deploy/setup-vps.sh   # JRE 17 + Ollama + модели (один раз)
./deploy/deploy.sh root@<vps>                     # fat jar + корпус + индекс + systemd
VERIFY_BASE_URL=http://<vps>:8030 VERIFY_TOKEN=<токен> ./run.sh verify
```

`deploy.sh` сам генерирует токен в `/opt/day30/.env` при первом деплое и
печатает его. Ollama на VPS слушает только localhost — наружу торчит один
порт сервиса, и на нём авторизация.
