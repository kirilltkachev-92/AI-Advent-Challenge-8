# День 25. Мини-чат с RAG + памятью (production-like)

CLI-чат, который: хранит историю диалога (с персистом на диск), на каждый вопрос ищет
контекст в базе через RAG, отвечает с учётом найденного и **всегда выводит источники**.
Усиление: «память задачи» — цель диалога, что уже уточнено, зафиксированные
ограничения и термины.

## Как устроено

- **`ChatSession`** — ядро: история диалога + память задачи, персист в
  `output/chat-session.json` (перезапустил CLI — диалог продолжается). Интерактивный чат
  и сценарные прогоны ходят через один и тот же код.
- **Роутер + conversation-aware rewrite** (`RagAgent.kt`): follow-up («а на переводах?»)
  переписывается в самостоятельный поисковый запрос с учётом истории и памяти задачи;
  мета-вопросы о самом диалоге («какая у нас цель?», «суммируй») отвечаются из памяти,
  без обращения к базе — у таких ответов источников нет по определению, и они помечаются.
- **Поиск** — воронка Дней 23–24: rewrite → top-20 по косинусу → порог 0.55 →
  LLM-реранкер → порог ≥6 → top-4; правила Дня 24 сохранены: ответы строго по чанкам,
  обязательные источники, «не знаю» + просьба уточнить при слабом контексте.
- **Память задачи** (`TaskState.kt`) — `{цель, уточнено, ограничения, термины}`,
  обновляется отдельным вызовом LLM после каждого хода и инжектится в каждый промпт.
  В окно попадают только последние `HISTORY_TURNS=8` ходов «сырыми» — всё важное
  из более старых ходов живёт в памяти задачи (поэтому цель не теряется на длинных диалогах).
- Команды чата: `/state` (память задачи), `/history`, `/new`, `/exit`.

## Проверка: 2 длинных сценария по 12 сообщений (output/scenarios.md)

| | сценарий 1: доклад о few-shot | сценарий 2: пост о рисках |
|---|---|---|
| ответы по базе с источниками | **7/7** | **8/8** |
| мета-ответы (из памяти диалога) | 5 | 4 |
| цель удержана к концу (судья) | **2/2** | **2/2** |

В сценариях есть всё «плохое», что бывает в живых диалогах: follow-up с местоимениями
(«а на переводах?»), смена ограничений на ходу («zero-shot не нужен»), фиксация терминов,
вопрос-ловушка мимо базы, возврат к теме после отвлечения, «напомни цель» и финальное резюме.

Наблюдения:

- **Цель не теряется**: в конце сценария 1 память задачи содержит цель, все зафиксированные
  ограничения (включая «сравниваем только few-shot с fine-tuned SOTA») и накопленные цифры;
  финальное резюме судья оценил 2/2 — «соблюдает ограничения (нет zero-shot, кратко, с цифрами)».
- **Ловушку («Какая столица Франции?») перехватил роутер**: вопрос ушёл по мета-ветке,
  и ассистент вежливо отказался, сославшись на цель диалога — не выдумал «Париж» от лица базы.
  Защита получилась двухуровневой: роутер отсекает офф-топик, порог реранка — вопросы
  «по теме, но без ответа в базе».
- **Каждый факт — с источниками**: 15/15 ответов по базе содержат валидные источники
  (chunk_id + section), проверено кодом через `Validator`.

## Запуск

```bash
brew install ollama && ollama serve      # если ещё не запущена
ollama pull nomic-embed-text             # модель эмбеддингов (~270 МБ)
cp .env.example .env                     # и вписать DEEPSEEK_API_KEY

./run.sh                 # интерактивный чат (индекс построится сам)
./run.sh scenarios       # 2 длинных сценария → output/scenarios.md
./run.sh index           # только построить индекс из docs/
./demo.sh                # демо для видео: мини-диалог + сценарии + отчёт
```

## Файлы

- `Main.kt` — CLI: интерактивный чат + сценарии (новое)
- `ChatSession.kt` — история диалога, персист, обновление памяти (новое)
- `TaskState.kt` — «память задачи» и её LLM-обновлятор (новое)
- `RagAgent.kt` — роутер мета/база + conversation-aware rewrite + правила Дня 24 (новое)
- `Scenarios.kt` — 2 длинных сценария, проверки, судья цели, отчёт (новое)
- из Дней 21–24: `Documents.kt`, `Chunking.kt`, `OllamaEmbedder.kt`, `IndexStore.kt`,
  `Search.kt`, `DeepSeekClient.kt`, `Reranker.kt`, `Validator.kt`
