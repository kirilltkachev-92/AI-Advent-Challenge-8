# День 21. Индексация документов

Пайплайн индексации документов для будущего RAG: **документы → текст → чанки
(2 стратегии) → эмбеддинги → локальный JSON-индекс с метаданными → сравнение стратегий**.

## Корпус (≈ 79 страниц)

Одна большая NLP-статья: **Brown et al., «Language Models are Few-Shot Learners» (GPT-3),
arXiv 2005.14165** — `docs/gpt3-language-models-are-few-shot-learners.pdf`,
237 100 символов текста ≈ 79 страниц (минимум задания — 20–30).

Корпус полностью автономен: индексируется только содержимое `docs/`
(поддерживаются pdf / md / kt — можно докинуть свои файлы).

## Пайплайн

1. **PDF → текст.** PDFBox (единственная «тяжёлая» зависимость), склейка
   переносов-дефисов на границах строк.
2. **Chunking — две стратегии:**
   - `fixed` — окно 1200 символов с перекрытием 200, откат до границы слова.
     Ничего не знает про структуру.
   - `structure` — по разделам статьи: нумерованные заголовки («3.2 Closed Book Question
     Answering»), Abstract/References, приложения с буквенной нумерацией («C Details of Test
     Set Contamination Studies», отличаем от обычной строки «A model can …» проверкой Title
     Case), строки оглавления отсеиваются по номеру страницы в конце. Разделы длиннее 4000
     символов дорезаются окном (контекст эмбеддера ограничен), но сохраняют `section`.
     Для markdown режет по `#…`, для кода — по top-level объявлениям.
3. **Эмбеддинги — локально**, Ollama `nomic-embed-text` (dim=768), протокол `/api/embed`
   написан вручную; батчи по 16, task-префиксы `search_document:` / `search_query:`,
   векторы нормируются при индексации (косинус = скалярное произведение).
4. **Индекс — JSON** (`output/index-fixed.json`, `output/index-structure.json`).
   Метаданные каждого чанка из задания: `source`, `title`, `section`, `chunk_id`
   (+ `strategy`, `char_start`/`char_end`) и вектор.
5. **Сравнение** — `output/comparison.md`: статистика чанков, retrieval на 6 тестовых
   вопросах по статье и LLM-судья (DeepSeek, опционально): какой из двух извлечённых
   контекстов лучше отвечает на вопрос.

## Результаты сравнения

| метрика | fixed | structure |
|---|---|---|
| чанков | 238 | 89 |
| средний размер | 1195 симв. | 2662 симв. |
| мин / макс | 882 / 1200 | 95 / 4000 |
| `section` в метаданных | нет | у всех |
| судья DeepSeek (6 вопросов) | 2 победы | **4** победы |

Выводы (все скоры — в `output/comparison.md`):

- **structure выигрывает, когда вопрос совпадает с темой раздела**: «what limitations…» →
  целиком раздел «5 Limitations», «data contamination…» → «4 Measuring and Preventing
  Memorization Of Benchmarks» + приложение C. Судья отдал ей 4 из 6 — контекст полнее,
  и метаданное `section` бесплатно даёт цитируемость источника.
- **fixed берёт точечные факты**: «сколько параметров у GPT-3» — ответ в одном абзаце,
  и маленькое сфокусированное окно бьёт большой раздел (у structure top-3 контекст
  раздувается до ~8К символов против стабильных ~3.5К).
- По top-1 косинусу стратегии близки; решает **гранулярность**: у fixed стабильный размер
  и плотность, у structure — полнота и адресуемость. Практичный гибрид: резать по
  структурным границам, длинные разделы дорезать окнами (здесь так и сделано, лимит 4000),
  метаданные секций хранить всегда.
- Качество structure упирается в детект заголовков: строки оглавления и буквенные
  приложения пришлось обрабатывать отдельно — на «грязных» PDF это главный источник ошибок.

## Запуск

```bash
brew install ollama && ollama serve      # если ещё не запущена
ollama pull nomic-embed-text             # модель эмбеддингов (~270 МБ)

./run.sh                 # весь пайплайн: index + compare
./run.sh index           # только построить индексы
./run.sh compare         # только сравнение (нужен DEEPSEEK_API_KEY для судьи, иначе без него)
./run.sh search "How many parameters does the largest GPT-3 model have?"
./demo.sh                # демо для видео: пайплайн + примеры поиска
```

Ключи не обязательны: эмбеддинги полностью локальные. `DEEPSEEK_API_KEY` в `.env`
включает LLM-судью в сравнении.

## Файлы

- `Main.kt` — CLI пайплайна (index / compare / search)
- `Documents.kt` — загрузка корпуса из `docs/`, pdf → текст
- `Chunking.kt` — обе стратегии chunking + метаданные чанков
- `OllamaEmbedder.kt` — эмбеддинги через Ollama, протокол вручную
- `IndexStore.kt` — JSON-индекс (сохранение/загрузка)
- `Search.kt` — косинусный поиск top-k
- `Comparison.kt` — отчёт сравнения стратегий
- `DeepSeekJudge.kt` — опциональный LLM-судья (протокол вручную)
