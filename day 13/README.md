# День 13 — кодинг-агент с задачей-конечным автоматом и роем агентов по этапам

Консольный **кодинг-агент** на Kotlin. Задача проходит по конечному автомату
`planning → execution → validation → done`; **на каждом этапе работают свои агенты**, а часть
этапов поднимает **рой параллельных инстансов**. Переходы между этапами **детерминированы и
проверяются в коде**, результат каждого этапа — артефакт, который передаётся дальше. Состояние
сохраняется на диск, поэтому работу можно поставить на паузу на любом этапе и продолжить без
повторных объяснений.

Дизайн следует разбору наставника (Алексей Гладков): «сделай 4 агента и переводи детерминированно
между ними работу, каждый отвечает за свой статус»; «на первом этапе запускаешь несколько агентов
параллельно, каждый по своему направлению»; «завершилась часть, родился артефакт — поехали в
следующую»; пауза = явное прерывание с сохранением стейта; сверху — агент-оркестратор.

## Конечный автомат и агенты этапов

```
 planning ─/next▶ execution ─/next▶ validation ─/next▶ done
    ▲                  │                  │
    └──────/back───────┘◀──────/back──────┘
```

| Этап | Агенты (несколько инстансов на этап) | Артефакт |
|------|--------------------------------------|----------|
| **planning** | `research:architecture` ∥ `research:risks` ∥ `research:libraries` → `planner` | план / ТЗ |
| **execution** | `coder` ∥ `test-writer` | код + тесты |
| **validation** | `review:correctness` ∥ `review:security` → `validator` | вердикт PASS/FAIL |
| **done** | `orchestrator` (надзор) | итоговая сборка |

`∥` — параллельный рой инстансов; `→` — агент-синтезатор сводит результаты роя в артефакт этапа.

Ревью различает **блокеры** (код не работает / не решает задачу / реальная уязвимость) и
необязательные **замечания** (стиль, улучшения); вердикт `FAIL` ставится только при блокерах.
Если ревью вернуло `FAIL`, команда `/back` (validation → execution) перезапускает кодера, **передав
ему вердикт ревью** — это цикл исправления; затем `/next` снова прогоняет проверку.

**Агент** (`Agent.kt`) — как класс в ЯП: имя, роль, свой system prompt, вход/выход через `run()`.
Из одного «класса» поднимается сколько угодно инстансов; рой запускается параллельно
(`Swarm.run`, пул потоков). Профиль разработчика подмешивается в каждого агента.

## Детерминированные переходы (в коде, не в промпте)

Разрешённые переходы — единственный источник истины (`TaskStage.transitions`):

| Этап | Куда можно |
|------|-----------|
| `planning` | `execution` |
| `execution` | `validation`, `planning` |
| `validation` | `done`, `execution` |
| `done` | — |

Недопустимый переход (например `execution → done`) **отклоняется кодом**, состояние не меняется,
показываются допустимые варианты. Это и есть детерминизм поверх недетерминированной модели.

## Пауза и продолжение

После каждого этапа пайплайн **останавливается** и ждёт решения пользователя (`/next` — принять,
`/back` — переделать, текст — замечания). Состояние и все артефакты лежат в `state/task.md`
(читаемый Markdown). Закрыли приложение на любом этапе — при следующем запуске `/state` покажет
тот же этап и готовые артефакты, а агенты следующего этапа получат их как вход: продолжение
**без повторных объяснений** (артефакты не пересчитываются).

## Персонализация (наследие дня 12)

Профиль разработчика (`profiles/dev.json`): язык, стек, уровень, подробность, жёсткие ограничения.
Задаётся коротким интервью при первом запуске и **подмешивается в каждого агента** — исследование,
план, код и ревью идут под ваш стек и правила.

## Команды

```
/task <запрос>     начать задачу (или просто напишите запрос, когда задачи нет)
/state             этап, ожидаемое действие, готовые артефакты, история переходов
/show <этап>       показать артефакт этапа (planning|execution|validation|done)
/next              принять этап и перейти к следующему (запускает его агентов)
/back              вернуться на предыдущий этап и перегенерировать его
/stage <id>        перейти на этап (валидируется)
текст во время паузы — замечания: текущий этап перегенерируется
/profile · /new-profile · /clear · /reset · /help · /exit
```

> Внимание: полный проход задачи делает ~10 вызовов LLM (planning 4 + execution 2 + validation 3 + done 1).

## Запуск

```bash
cd "day 13"
cp .env.example .env     # впишите DEEPSEEK_API_KEY
./run.sh
```

## Тесты

```bash
./gradlew test
```

13 тестов: таблица переходов, прямой путь и `/back`, отклонение недопустимого перехода без
изменения состояния, артефакты по этапам, snapshot/restore (пауза/возобновление); многоагентность —
planning поднимает рой из 3 исследователей + планировщик, execution запускает coder ∥ test-writer,
validation — рой из 2 ревьюеров + валидатор, профиль подмешивается в каждого агента; оркестрация —
`startTask` прогоняет planning, `advance` запускает следующий этап, недопустимый переход блокируется,
состояние переживает перезапуск.
