# День 19. Композиция MCP-инструментов

**Один** MCP-сервер с **тремя инструментами** (`search` → `summarize` → `save_to_file`)
и агент на DeepSeek, который из обычного запроса пользователя САМ выстраивает цепочку
вызовов и передаёт результат одного инструмента в следующий.

Продолжение Дней 16–18: тот же **рукописный MCP** (JSON-RPC 2.0 поверх Streamable HTTP,
без SDK). Учтены уточнения Алексея: *«не надо делать 3 разных mcp»* (всё на одном сервере),
*«все на mcp»* (инструменты на сервере, не в агенте), *«суммаризация — это просто отчёт/итог»*
(никакого LLM на сервере — `summarize` детерминированно собирает конспект), и главное —
*«без хардкода в агенте, чтобы их вызывала LLM»*: порядок инструментов выбирает модель.

## Пайплайн

```
Запрос: «Найди статьи про Эрмитаж и сохрани конспект в файл hermitage.md»

  search(query)            ── ПОЛУЧАЕТ ДАННЫЕ  → список заголовков статей (Википедия)
        │ titles
        ▼
  summarize(topic, titles) ── ОБРАБАТЫВАЕТ     → markdown-конспект из вводных абзацев
        │ content
        ▼
  save_to_file(filename,   ── СОХРАНЯЕТ         → output/hermitage.md
              content)
```

Данные текут через агента: вывод `search` → аргумент `summarize` → аргумент `save_to_file`.
Последовательность НЕ захардкожена — её строит LLM по запросу (function calling в цикле).

## Что сделано по заданию

| Требование задания | Где в коде |
|---|---|
| Несколько MCP-инструментов (один сервер) | `McpServer.compositionServer(...)` в [`McpServer.kt`](src/main/kotlin/McpServer.kt) |
| 👉 первый получает данные | `search` → [`WikiApi.kt`](src/main/kotlin/WikiApi.kt) (поиск по Википедии) |
| 👉 второй обрабатывает | `summarize` → [`Report.kt`](src/main/kotlin/Report.kt) (конспект-итог, без LLM) |
| 👉 третий сохраняет | `save_to_file` → [`NoteSaver.kt`](src/main/kotlin/NoteSaver.kt) (запись в `output/`) |
| 👉 автоматическое выполнение цепочки | [`PipelineAgent.kt`](src/main/kotlin/PipelineAgent.kt) — LLM сам строит цепочку |
| 👉 корректность передачи данных | тест «вся цепочка проходит…»: файл == результат `summarize` |

## Инструменты

| Инструмент | Параметры | Что делает |
|---|---|---|
| `search`        | `query` (req), `limit` | ищет статьи, возвращает заголовки + сниппеты |
| `summarize`     | `titles` (req, array), `topic` | собирает markdown-конспект по заголовкам |
| `save_to_file`  | `filename` (req), `content` (req) | сохраняет текст в файл (имя санируется) |
| `save_articles` | `titles` (req, array) | сохраняет ПОЛНЫЙ оригинал каждой статьи в ОТДЕЛЬНЫЙ файл (без суммаризации) |

Две ветки цепочки выбирает сама LLM по запросу:

- «сохрани **конспект**» → `search → summarize → save_to_file` (один итоговый файл);
- «сохрани **оригинал** каждой статьи по отдельности» → `search → save_articles` (по файлу на статью, текст как есть). Контраст этих двух эффектов виден в `output/` после `./demo.sh`.

## Запуск

```bash
cp .env.example .env      # впишите DEEPSEEK_API_KEY (или подхватится из .env прошлых дней)
./run.sh
```

Пример:

```
you> Найди статьи про Эрмитаж и сохрани конспект в файл hermitage.md
→ Агент строит цепочку…
Цепочка вызовов MCP:
  1. ⚙ search {"query":"Эрмитаж","limit":5}      ← Найдено статей: 5 …
  2. ⚙ summarize {"topic":"Эрмитаж","titles":[…]} ← # Конспект: Эрмитаж …
  3. ⚙ save_to_file {"filename":"hermitage.md",…} ← Сохранено в output/hermitage.md (4508 байт).
────────────────────────────────────────────────────────────
Готово! Нашёл 5 статей, собрал конспект и сохранил в output/hermitage.md.
```

Команды REPL: `<запрос>` (главный сценарий), `tool <name> <json>` (прямой вызов без LLM),
`list`, `quit`.

### Видео-демо одной командой

```bash
./demo.sh                 # один запрос → автоцепочка → показывает созданный файл
```

## Конфигурация (`.env` / окружение)

- `DEEPSEEK_API_KEY` — ключ агента (без него доступен ручной `tool`-вызов цепочки);
- `WIKI_LANG` — язык Википедии (по умолчанию `ru`);
- `WIKI_DELAY_MS` — пауза между запросами к Википедии, мс (по умолчанию `500`); при 429/503 клиент сам ждёт `Retry-After` и повторяет запрос с нарастающей паузой;
- `OUTPUT_DIR` — каталог для `save_to_file` (по умолчанию `output`);
- `MCP_PORT` / `MCP_SERVER_URL` — порт встроенного или адрес внешнего MCP-сервера.
