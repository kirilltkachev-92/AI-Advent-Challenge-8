# Архитектура проекта AI Advent Challenge #8

## Что это за проект

Репозиторий — марафон из 31 ежедневного задания по работе с LLM.
Каждый день — самостоятельный Kotlin/Gradle-проект в папке `day N` со своим
`README.md`, `run.sh` и исходниками в `src/main/kotlin`. Общий принцип
марафона: все протоколы (HTTP API моделей, MCP, парсеры) пишутся вручную,
без фреймворков и SDK — чтобы протокол был виден целиком.

## Структура репозитория

- `day 1` … `day 15` — базовые приёмы: промпты, диалоги, структурированные
  ответы, сравнение моделей.
- `day 16` — MCP-клиент к публичному серверу Microsoft Learn (JSON-RPC 2.0
  поверх Streamable HTTP).
- `day 17` — собственный MCP-сервер погоды + агент DeepSeek с function calling.
- `day 18`–`day 20` — агенты и мультиагентные сценарии поверх MCP.
- `day 21`–`day 25` — Неделя 5: RAG-пайплайн (чанки → эмбеддинги →
  JSON-индекс → top-K поиск), эксперименты со стратегиями чанкинга.
- `day 26`–`day 30` — Неделя 6: локальные модели через Ollama, сравнение
  локальной и облачной генерации, RAG по истории чата марафона,
  приватный LLM-сервис с токенами и лимитами.
- `day 31` — этот проект: ассистент разработчика, понимающий репозиторий.

## Стек

- Kotlin 2.0 (JVM 17), Gradle, kotlinx-serialization для JSON.
- Облачная генерация — DeepSeek (`/chat/completions`, OpenAI-совместимый).
- Эмбеддинги — локальная Ollama, модель `nomic-embed-text`.
- Никаких HTTP-фреймворков: сервер — `com.sun.net.httpserver` из JDK,
  клиенты — `java.net.http.HttpClient`.

## Как устроен день 31

Три слоя, каждый в своём файле `src/main/kotlin`:

1. **RAG по документации** — `DocsCorpus.kt` собирает все `README.md`
   репозитория и папку `day 31/docs`, режет на чанки по заголовкам;
   `Indexer.kt` считает эмбеддинги и кладёт индекс в
   `output/index-docs.json`; `Search.kt` ищет top-K по косинусу с
   бонусами за точные слова.
2. **MCP-сервер git** — `GitMcp.kt` поднимает JSON-RPC 2.0 поверх
   Streamable HTTP на localhost и отдаёт инструменты `git_branch`,
   `git_status`, `git_files`, `git_diff`, `git_log`; каждый инструмент —
   обёртка над настоящей командой `git` в корне репозитория.
3. **Ассистент** — `Assistant.kt`: DeepSeek с function calling получает
   список MCP-инструментов из `tools/list`, фрагменты документации из
   RAG и отвечает на `/help`-вопросы, при необходимости дёргая git через
   MCP (`tools/call`).
