# День 22. Первый RAG-запрос

Функция дня: **вопрос → поиск релевантных чанков → объединение с вопросом → запрос к LLM**.
Агент с двумя режимами (с RAG / без RAG) + 10 контрольных вопросов и сравнение качества.

## Как устроено

- **База** — статья GPT-3 (Brown et al., «Language Models are Few-Shot Learners»,
  arXiv 2005.14165, ~79 страниц) в `docs/`. День полностью автономен: пайплайн индексации
  Дня 21 (PDF → текст → структурные чанки → эмбеддинги Ollama `nomic-embed-text` →
  JSON-индекс с метаданными) скопирован сюда и строит индекс локально.
  Стратегия по умолчанию — `structure` (победитель Дня 21), переключается `RAG_STRATEGY`.
- **Агент** (`RagAgent.kt`) — один LLM (DeepSeek, temperature 0), два режима:
  - **без RAG** — вопрос уходит модели как есть;
  - **с RAG** — вопрос → эмбеддинг запроса → top-4 чанков по косинусу → чанки с метками
    `[section]` + вопрос → модель. Промпт требует отвечать только по контексту, цитировать
    разделы и честно говорить, если ответа в контексте нет.
- **Контрольный набор** (`ControlSet.kt`) — 10 вопросов по базе, для каждого зафиксированы
  **ожидание** (что должно быть в ответе) и **ожидаемые источники** (разделы статьи).
- **Сравнение качества** (`Evaluation.kt`) — оба режима отвечают на все 10 вопросов;
  судья (тот же DeepSeek, отдельный промпт) вслепую ставит каждому ответу 0–2 балла
  относительно ожидания; для RAG автоматически проверяется, попали ли ожидаемые разделы
  в извлечённые чанки. Отчёт — `output/evaluation.md`.

## Результаты

| | без RAG | с RAG |
|---|---|---|
| баллы судьи (макс 20) | 16 | **18** |
| ответов на 2 балла | 8 | 9 |
| ответов на 0 баллов | 2 | 1 |
| попадание в ожидаемые источники | — | 12 из 15 |

Что показало сравнение (детали — в `output/evaluation.md`):

- **RAG чинит выдуманные цифры.** Без RAG модель уверенно назвала неверные проценты на
  LAMBADA (76.2% вместо 86.4%, Q4) и неверный паттерн внимания (Q10) — с RAG оба ответа
  точны, с цитатами разделов. Это главный выигрыш: статья знаменитая и модель «помнит» её
  в общих чертах, но точечные факты перевирает.
- **Единственный ноль RAG — промах retrieval, а не генерации (Q2).** Вопрос про состав
  обучающей выборки: нужный раздел «2.2 Training Dataset» занял лишь 24-е место по косинусу —
  длинная секция в основном про методику фильтрации Common Crawl, и точечный факт (таблица
  с весами) «растворился» в эмбеддинге. Агент честно ответил «в контексте этого нет»
  (правильное поведение по промпту), а вот модель без RAG ответила верно из памяти.
  Ровно та слабость структурной стратегии, которую показал День 21: точечные факты лучше
  берут маленькие fixed-окна.
- **Судья вслепую** (не знает, какой режим отвечал) — оценка сравнима между режимами;
  автопроверка источников делает «какие источники должны быть использованы» измеримым:
  12 из 15 ожидаемых разделов реально попали в top-4.

## Запуск

```bash
brew install ollama && ollama serve      # если ещё не запущена
ollama pull nomic-embed-text             # модель эмбеддингов (~270 МБ)
cp .env.example .env                     # и вписать DEEPSEEK_API_KEY (нужен агенту и судье)

./run.sh                 # индекс (если ещё нет) + сравнение на 10 вопросах
./run.sh index           # только построить индекс из docs/
./run.sh ask "What accuracy does GPT-3 achieve on LAMBADA few-shot?"   # оба режима бок о бок
./run.sh eval            # 10 контрольных вопросов + судья → output/evaluation.md
./demo.sh                # демо для видео
```

## Файлы

- `Main.kt` — CLI (index / ask / eval)
- `RagAgent.kt` — агент с двумя режимами (суть дня)
- `ControlSet.kt` — 10 контрольных вопросов: ожидание + ожидаемые источники
- `Evaluation.kt` — прогон обоих режимов, слепой судья, отчёт
- `DeepSeekClient.kt` — /chat/completions вручную
- из Дня 21: `Documents.kt`, `Chunking.kt`, `OllamaEmbedder.kt`, `IndexStore.kt`, `Search.kt`
