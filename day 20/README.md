# День 20. Orchestration MCP

**Четыре** отдельных MCP-сервера разной «природы», агент-оркестратор на DeepSeek и
**роутер**, который сводит их инструменты в один список и маршрутизирует каждый вызов на
сервер-владельца. Из обычного запроса агент САМ выбирает нужные инструменты с РАЗНЫХ серверов
и строит длинный кросс-серверный флоу.

Продолжение Дней 16–19: тот же **рукописный MCP** (JSON-RPC 2.0 поверх Streamable HTTP, без
SDK). Учтены уточнения Алексея к Дню 20: *«да нужно прям большой сделать»*, *«желательно вообще
соркестрировать 3–4 mcp»* (здесь ровно 4 сервера), и общее правило *«не использовать ничего
модельспецифичного»* — выбор инструментов идёт стандартным function calling в цикле, протокол
MCP виден целиком, ничего DeepSeek-специфичного в маршрутизации нет.

## Четыре MCP-сервера

| Сервер | Природа | Порт | Инструменты |
|---|---|---|---|
| `research-mcp` | источник знаний (Википедия) | 8781 | `wiki_search`, `wiki_extract` |
| `weather-mcp`  | источник погоды (Open-Meteo) | 8782 | `geocode`, `get_weather` |
| `report-mcp`   | обработка текста (без LLM) | 8783 | `compose_report`, `word_count` |
| `storage-mcp`  | файловое хранилище | 8784 | `save_to_file`, `list_files`, `read_file` |

## Оркестрация

```
Запрос: «Собери досье по Санкт-Петербургу (справка + погода) и сохрани в spb.md»

  [research-mcp] wiki_search → wiki_extract   ─ справка из Википедии
                                  │
  [weather-mcp]  geocode → get_weather        ─ текущая погода
                                  │  (оба раздела с РАЗНЫХ серверов)
                                  ▼
  [report-mcp]   compose_report(title, [справка, погода]) ─ markdown-отчёт
                                  │ content
                                  ▼
  [storage-mcp]  save_to_file(spb.md, content) → list_files ─ сохранить и подтвердить
```

**Роутер** ([`McpRouter.kt`](src/main/kotlin/McpRouter.kt)) держит таблицу `tool → сервер`:
агент видит плоский список инструментов, а доставку вызова на нужный сервер обеспечивает роутер.
Последовательность и выбор НЕ захардкожены — их строит LLM по запросу (function calling в цикле,
[`OrchestratorAgent.kt`](src/main/kotlin/OrchestratorAgent.kt)).

## Что сделано по заданию

| Требование задания | Где в коде |
|---|---|
| Зарегистрировать несколько MCP-серверов | `Config.servers()` + `Boot.startAllServers(...)` — 4 сервера |
| 👉 агент выбирает нужный инструмент | `OrchestratorAgent` отдаёт LLM все инструменты, `tool_choice=auto` |
| 👉 корректно маршрутизирует запросы | `McpRouter.callTool` → клиент сервера-владельца (таблица `tool→сервер`) |
| 👉 длинный флоу взаимодействия | сценарий «досье»: 6–8 вызовов через 4 сервера |
| 👉 инструменты с РАЗНЫХ серверов | research + weather + report + storage в одном флоу |
| 👉 корректность выбора и порядка | тест «длинный кросс-серверный флоу…»: файл == отчёт |

## Запуск

```bash
cp .env.example .env      # впишите DEEPSEEK_API_KEY (или подхватится из .env прошлых дней)
./run.sh
```

Пример (реальный прогон):

```
✓ Поднято MCP-серверов: 4
   • research-mcp → http://localhost:8781/mcp
   • weather-mcp  → http://localhost:8782/mcp
   • report-mcp   → http://localhost:8783/mcp
   • storage-mcp  → http://localhost:8784/mcp

you> Собери досье по Санкт-Петербургу: справка и погода, сохрани в spb.md, покажи список файлов
Флоу вызовов MCP (серверов задействовано: 4 — research-mcp, weather-mcp, report-mcp, storage-mcp):
  1. [research-mcp] ⚙ wiki_search {"query":"Санкт-Петербург"}      ← Найдено статей: 1 …
  2. [weather-mcp]  ⚙ geocode {"city":"Санкт-Петербург"}            ← 59.94, 30.31 …
  3. [weather-mcp]  ⚙ get_weather {"city":"Санкт-Петербург"}        ← пасмурно, 17.8°C …
  4. [research-mcp] ⚙ wiki_extract {"title":"Санкт-Петербург"}      ← # Санкт-Петербург …
  5. [report-mcp]   ⚙ compose_report {"title":"Досье…","sections":[…]} ← # Досье: Санкт-Петербург …
  6. [storage-mcp]  ⚙ save_to_file {"filename":"spb.md",…}          ← Сохранено в output/spb.md (4152 байт).
  7. [storage-mcp]  ⚙ list_files {}                                 ← Сохранённых файлов: 1. • spb.md
```

Команды REPL: `<запрос>` (главный сценарий), `tool <name> <json>` (прямой вызов через роутер,
видно маршрут), `list` (инструменты по серверам), `quit`.

### Видео-демо одной командой

```bash
./demo.sh                 # два длинных кросс-серверных флоу + чтение файла обратно
```

### Только серверы (MCP Inspector)

```bash
./run-server.sh           # все 4 сервера: localhost:8781…8784/mcp
# npx @modelcontextprotocol/inspector → Streamable HTTP → любой из URL
```


[`McpOrchestrationTest.kt`](src/test/kotlin/McpOrchestrationTest.kt) (заглушки вместо сети, без
LLM): поднимает все 4 сервера на тестовых портах и подключает к ним роутер по реальному
HTTP/JSON-RPC — проверяет сведение инструментов со всех серверов, **таблицу маршрутизации**
(`tool → правильный сервер`), ошибку маршрута для несуществующего инструмента и **длинный
кросс-серверный флоу** с передачей данных без искажения (файл == результат `compose_report`).

## Конфигурация (`.env` / окружение)

- `DEEPSEEK_API_KEY` — ключ агента (без него доступен ручной `tool`-вызов через роутер);
- `WIKI_LANG` — язык Википедии (по умолчанию `ru`);
- `WIKI_DELAY_MS` — пауза между запросами к Википедии, мс (по умолчанию `500`);
- `OUTPUT_DIR` — каталог для `save_to_file` (по умолчанию `output`);
- `MCP_BASE_PORT` — базовый порт; серверы занимают `base..base+3` (по умолчанию `8781`).
