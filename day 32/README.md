# День 32. Автоматизация ревью кода

**Автоматическое AI-ревью PR**: реактивный пайплайн GitHub Actions
([`.github/workflows/ai-review.yml`](../.github/workflows/ai-review.yml)) —
на каждый pull request ассистент получает diff и список изменённых файлов,
достаёт контекст из RAG-индекса **документации и кода** всего репозитория
и оставляет ревью комментом прямо в PR: потенциальные баги, архитектурные
проблемы, рекомендации.

По обсуждению в чате: тьютор уточнил, что нужно именно **реактивное действие**
(триггер «создан PR», не запуск по команде), что возвращать ревью можно «прям
в гитхаб — в PR в виде комментов», и что перестраивать RAG на каждый прогон
«как-то очень избыточно» — поэтому индекс строится локально один раз,
коммитится в репозиторий и в CI только сверяется по SHA корпуса
(на CPU runner-а полная пересборка занимает десятки минут — проверено).

## Как устроено

```
PR открыт/обновлён ──on: pull_request──▶ GitHub Actions (ubuntu, Java 17, Ollama)
    │                                        │ кеш: модель эмбеддингов + RAG-индекс
    ▼                                        ▼
GITHUB_EVENT_PATH ──▶ номер PR, base/head ──▶ git diff base...head (+ файлы)
                                                  │
README + docs + все *.kt (991 чанк) ──top-K──▶ контекст: конвенции и похожий код
                                                  │
system(ревьюер) + файлы + RAG + diff ──▶ DeepSeek ──▶ ## 🐛 баги
                                                      ## 🏛 архитектура
                                                      ## 💡 рекомендации
                                                  │
POST /repos/{repo}/issues/{pr}/comments ──▶ 🤖 коммент в PR
```

- **Diff и изменённые файлы** (`GitDiff.kt`): three-dot `git diff base...head`
  по SHA из события pull_request; в Actions репозиторий выкачан с
  `fetch-depth: 0`. Длинный diff обрезается под бюджет (40k символов).
- **RAG = документация + код** (`RepoCorpus.kt`): markdown режется по разделам
  (как в Дне 31), Kotlin-исходники — окнами по 60 строк с путём файла в
  заголовке. Retrieval — запрос на каждый изменённый файл (путь + добавленные
  строки) плюс общий по списку файлов; хиты дедуплицируются и разворачиваются
  соседними чанками. Так ревьюер видит конвенции марафона и похожий код
  других дней.
- **Индекс не строится заново на каждый PR** (`Indexer.kt`):
  `output/index-repo.json` (11 МБ) построен локально и закоммичен; внутри —
  SHA-256 корпуса, и CI пересчитывает эмбеддинги только если документация
  или код реально поменялись, а индекс не переиндексировали. После больших
  изменений — `./run.sh index` и закоммитить обновлённый файл.
- **Ревью** (`Reviewer.kt`): DeepSeek, temperature 0; промпт требует три секции
  из задания, ссылки `файл:строка` и честное «не найдено» вместо надуманных
  замечаний. **Комментарий в PR** (`GitHubClient.kt`): GitHub REST API вручную
  на `java.net.http`, токен — штатный `GITHUB_TOKEN` воркфлоу с
  `permissions: pull-requests: write`.
- Протоколы вручную, как весь марафон: клиенты DeepSeek, Ollama и GitHub API —
  `java.net.http`, без фреймворков и SDK.

## Режимы

```bash
cd "day 32"
./run.sh index                  # построить/обновить индекс документации и кода
./run.sh review <base> [head]   # локальное ревью диффа → консоль + output/report.md
./run.sh ci                     # режим Actions: событие PR → ревью → коммент
```

## Чеклист задания

- ✅ **пайплайн запускается на PR** — `on: pull_request`, реактивно;
  устаревшие прогоны отменяются (`concurrency`)
- ✅ **ассистент получает diff** — и список изменённых файлов из события PR
- ✅ **использует RAG (документация + код)** — 30+ README, docs и 321
  Kotlin-файл в индексе; источники контекста перечислены в подписи коммента
- ✅ **возвращает баги / архитектурные проблемы / рекомендации** — три
  обязательные секции + вердикт, комментом в PR

Локальный прогон того же пайплайна (ревью диффа Дня 31) —
[`output/report.md`](output/report.md).

## Подключение к репозиторию

1. Секрет: `Settings → Secrets and variables → Actions → New repository secret`,
   имя `DEEPSEEK_API_KEY`.
2. Workflow уже в `.github/workflows/ai-review.yml` на ветке `main`.
3. Открыть любой PR — через пару минут под ним появится коммент
   «🤖 AI-ревью (день 32)». Для видео: `./demo.sh` создаёт демо-PR
   с заведомо проблемным кодом.
